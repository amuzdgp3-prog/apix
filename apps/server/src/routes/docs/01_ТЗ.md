## 1. Цель проекта

Разработать прогрессивное веб-приложение (PWA) для учёта работы игровых аппаратов типа «хватайка». Система заменяет существующее решение и обеспечивает:

- Работу до 15 техников и 2 администраторов одновременно.
- Автономный ввод обслуживаний (включая фото) при отсутствии интернета.
- Фоновую синхронизацию черновиков с сервером после восстановления связи.
- Полный пересчёт цепочек обслуживаний только на сервере.
- Аналитические отчёты для администраторов (только онлайн).
- Журнал аудита всех изменений.

## 4. Ключевые концепции и механика

### 4.1. Главная сущность — точка (локация)

Физическая точка (адрес) — это вечная сущность, к которой привязана вся история обслуживаний. Конкретный физический аппарат — это то, что стоит на точке в данный момент и может меняться.

**Концепция сессии (placement):**
- `location` — точка/адрес, существует всегда.
- `machine` — физический аппарат с номером на корпусе.
- `machine_placement` — сессия: какой аппарат стоит на какой точке с какого по какое число.
- Все обслуживания привязаны к `placement_id`, а не к аппарату напрямую.
- Пересчёт цепочки никогда не пересекает границу placement.

**Замена аппарата** (редкая операция, только через админку):
1. Администратор нажимает «Заменить аппарат» в карточке точки.
2. Вводит новый номер корпуса и начальные значения счётчиков.
3. Система закрывает текущий placement (`ended_at = now()`), создаёт новый.
4. Техники этого не видят — для них ничего не меняется.

### 4.2. Иерархия локаций

Дерево произвольной глубины — каждый узел может быть родителем любого другого. Глубина не фиксирована (не страна→регион→город→район, а просто дерево). Примеры узлов: город, район, торговый центр, улица, точка.

### 4.3. Маршруты

Именованный упорядоченный список точек (локаций). Не привязан жёстко к технику — разные техники могут обслуживать один маршрут в разные дни.

### 4.4. Игрушки на аппарате

Двухуровневое наследование:
1. **Тип аппарата** → базовый набор игрушек.
2. **Конкретный аппарат** → наследует набор от типа, можно добавить или убрать игрушки.

При обслуживании техник вводит, сколько штук каждой игрушки он **доложил** (не итоговое количество в аппарате).

### 4.5. Цены и себестоимость

- **Цена игры** — задаётся на аппарате, меняется редко. При синхронизации сервер делает снапшот текущей цены в запись обслуживания.
- **Закупочная цена игрушки** — единая для всех аппаратов, меняется часто. При синхронизации сервер делает снапшот текущей цены каждой игрушки. Изменение цены не пересчитывает прошлые обслуживания.

### 4.6. Финансовые расчёты

Все расчёты выполняются только на сервере при синхронизации.

```
new_games     = game_counter_current - game_counter_prev - test_games_current
revenue       = new_games × price_per_game                    (снапшот цены)
cost_of_toys  = Σ (qty_i × price_i)                          (снапшот цен)
roi           = revenue / prev.cost_of_toys                   (предыдущее обслуживание)
period_days   = (date_current - date_prev)                    (исключая простои)
```

- ROI первого обслуживания на placement = NULL.
- Если предыдущее обслуживание имело `is_operational = false` — период помечается как простой, выручка и ROI за этот период = NULL в аналитике.

### 4.7. Статус аппарата

Техник при каждом обслуживании отмечает статус: **работает / не работает**.

- Период между «не работает» и следующим «работает» = простой.
- Простои исключаются из аналитики (не портят средние показатели).
- В отчётах простои показываются отдельно.

### 4.8. Пересчёт цепочки (сервер)

Запускается при: добавлении обслуживания, удалении обслуживания, вставке задним числом.

Алгоритм:
1. Получить все обслуживания данного placement с `date >= start_date`, сортировка по `(date, service_time)`.
2. Получить предыдущее обслуживание перед `start_date`.
3. Для каждого обслуживания в цикле пересчитать: `period`, `new_games`, `new_prize`, `revenue`, `roi`, `before_service_id`.
4. Обновить записи в БД.
5. Выполняется в одной транзакции с `SELECT FOR UPDATE` по строкам placement.

Подробная реализация — см. [раздел 14](#14-алгоритм-пересчёта-цепочки).

### 4.9. Интервалы обслуживания и уведомления

На каждой точке/маршруте/районе задаётся диапазон: `min_days` и `max_days`. Конкретный аппарат может переопределить значения. Наследование: район → маршрут → аппарат.

Логика:
- Дней с последнего обслуживания < `min_days` — рано, ничего не показываем.
- В диапазоне — всё хорошо.
- Превысил `max_days` — аппарат «забытый».

Уведомление — ненавязчивый баннер снизу/сверху экрана при открытии приложения. Показывает количество забытых аппаратов и кнопку «Перейти к списку». Техник видит только свои забытые аппараты.

## 6. Функциональные требования

### 6.1. Справочники (только для админов, онлайн)

- **Типы аппаратов** — наименование + базовый набор игрушек.
- **Игрушки** — наименование, закупочная цена (рубли с копейками), история изменения цен.
- **Сотрудники** — email (логин), полное имя, пароль (bcrypt), роль, город, активен/неактивен.
- **Локации** — дерево произвольной глубины (название, родитель, тип узла), интервал обслуживания (`min_days`, `max_days`).
- **Маршруты** — наименование, описание, упорядоченный список точек, интервал обслуживания.
- **Аппараты** — см. п. 6.2.

### 6.2. Управление аппаратами (CRUD для админов)

Поля аппарата:
- `number` — номер на корпусе (целое, уникальный идентификатор).
- `location_id` — привязка к точке в дереве локаций.
- `type_id` — тип аппарата (наследует базовый набор игрушек).
- `status` — активен/неактивен.
- `price_per_game` — цена одной игры (рубли с копейками).
- `has_prize_counter` — флаг наличия счётчика призов (boolean).
- `game_counter_initial` — начальный счётчик игр (при первой установке).
- `prize_counter_initial` — начальный счётчик призов.
- `min_service_days` / `max_service_days` — переопределение интервала обслуживания (опционально).

Вкладки в карточке аппарата:
- **«Игрушки»** — унаследованные от типа + индивидуальные добавления/удаления.
- **«Маршруты»** — привязка к маршрутам.
- **«Техники»** — список привязанных техников.
- **«История»** — список placements (сессий) с датами и номерами корпусов.
- **«QR-код»** — кнопка «Распечатать QR» → генерирует PDF с QR-кодом для наклейки на аппарат.

Удаление аппарата — мягкое (`deleted_at`). Все placements и обслуживания остаются.

### 6.3. Обслуживание — ввод данных (техник)

**Выбор аппарата:**
- Поиск по номеру или адресу с автодополнением из кэша.
- Сканирование QR-кода камерой.

**Форма обслуживания (один аппарат = одно обслуживание в день):**

Блок «Прошлое обслуживание» (из кэша, только чтение):
- Дата, время, счётчик игр, новых игр, выручка, себестоимость игрушек.

Блок «Дата и время»:
- Дата (по умолчанию сегодня, можно изменить для ввода задним числом).
- Время (ЧЧ:ММ, по умолчанию текущее).

Блок «Счётчики»:
- Текущий счётчик игр (целое, обязательное). Под полем — прошлое значение.
- Тестовые игры (целое, по умолчанию 0). Под полем — прошлое значение.
- Счётчик призов (целое, необязательное) — показывается только если `has_prize_counter = true`.

Блок «Тренд ROI» (показывается, если есть хотя бы 1 прошлое обслуживание, данные из кэша):
- Мини-график последних до 5 значений ROI.
- Текущее значение ROI + бейдж тренда (растёт/падает/мало данных).
- Цветовая индикация: 🟢 ≥3.0, 🟡 2.0–2.9, 🔴 <2.0.
- При 1 обслуживании — цвет без тренда. При 2 — направление с пометкой «мало данных». При 3+ — полный тренд.

Блок «Предварительный расчёт» (только онлайн, запрос к серверу при вводе счётчика):
- Прирост игр, период обслуживания, среднее игр в день, выручка, ROI.

Блок «Статус аппарата»:
- Переключатель: Работает / Не работает.

Блок «Игрушки» (список разрешённых для этого аппарата):
- Для каждой игрушки: название, поле количества (целое, по умолчанию 0), цена (только чтение из справочника). Под полем — прошлое значение.
- Итоговая сумма затрат на игрушки.

Блок «Фото» (3 обязательных слота с подписями):
- 📷 Счётчик игр.
- 📷 Аппарат ДО обслуживания.
- 📷 Аппарат ПОСЛЕ обслуживания.
- Сжатие: до 1600px по большей стороне, качество 80%, целевой размер ≤500 КБ.

Блок «Комментарий» — текстовое поле (необязательное).

Кнопка «Сохранить в черновики» — сохраняет в IndexedDB, форма очищается.

### 6.4. Черновики и синхронизация

**Экран «Черновики»:**
- Список черновиков: номер аппарата, адрес, дата, статус («ожидает» / «ошибка»).
- Для каждого — кнопки «Редактировать» и «Удалить».
- Кнопка «Синхронизировать все».

**Алгоритм синхронизации:**
1. Клиент отправляет черновик на `POST /api/sync` (multipart/form-data: поля + 3 фото).
2. Сервер проверяет:
   - Существование placement для аппарата.
   - Монотонность счётчиков: `prev.game_counter ≤ current.game_counter ≤ next.game_counter` (если есть соседние обслуживания).
   - Допустимость даты/времени (не будущее).
   - Не более одного обслуживания в день для данного placement.
3. При успехе:
   - Сохраняет 3 фото в MinIO по пути: `{location_id}/{placement_id}/{year}/{month}/{service_id}_{тип}.jpg`.
   - Делает снапшоты цен (цена игры, цены игрушек).
   - Вставляет запись в `service` и `toy_distribution`.
   - Запускает пересчёт цепочки начиная с даты текущего обслуживания.
   - Обновляет кэш последних 5 обслуживаний для техника.
   - Записывает аудит.
   - Возвращает `{ success: true, recordId, calculations: {...} }`.
4. При ошибке — `{ success: false, errors: [...] }`.
5. Клиент удаляет успешные черновики, ошибочные помечает как `error`.

**Автоматическая синхронизация:**
- При открытии приложения с интернетом — автоматически обновляются справочники и кэш последних 5 обслуживаний.
- При событии `online`, если есть черновики — предлагается синхронизировать.
- iOS: Background Sync недоступен, синхронизация только при открытом приложении.

### 6.5. Кэширование на устройстве техника (IndexedDB)

При каждом открытии приложения с интернетом автоматически обновляется:
- Полный справочник аппаратов своего города (номер, адрес, тип, игрушки, техники).
- Дерево локаций.
- Справочник игрушек с ценами.
- Последние 5 обслуживаний по каждому закреплённому аппарату (для тренда ROI и прошлых значений).

При первом входе — полная загрузка. При последующих — только изменения с момента `last_sync` (по полю `updated_at`).

### 6.6. Уведомления о забытых аппаратах

При открытии приложения — проверка забытых аппаратов (превышен `max_service_days`). Если есть — баннер вверху/внизу экрана: «X аппаратов не обслуживались вовремя» + кнопка «Открыть список». Техник видит только свои аппараты. Администратор видит все.

**Вкладка «Забытые аппараты»:**
- Список: номер, адрес, последнее обслуживание, дней прошло, максимум дней.
- Сортировка по количеству дней просрочки.

## 8. Аналитика и отчёты

> Только онлайн, только администраторы.

### 8.1. Главный дашборд

Таблица всех активных аппаратов, отсортированная по выручке за последние 30 дней (от большей к меньшей).

Колонки:
- Номер аппарата, адрес.
- Текущий счётчик игр (из последнего обслуживания).
- Статус (работает/не работает).
- Выручка за 30 дней.
- Последний ROI + цветовой индикатор (🟢 ≥3.0, 🟡 2.0–2.9, 🔴 <2.0, ⚫ нет данных).
- Тренд ROI (мини-график последних 5 обслуживаний).
- Бейдж тренда: растёт / падает / нестабильно / мало данных / нет данных.

Правила тренда:
- 0 ROI значений — ⚫ нет данных.
- 1 значение — цвет без тренда.
- 2 значения — направление + «мало данных».
- 3+ значения — полный тренд.

### 8.2. Карточка аппарата

Верхняя зона (краткая статистика):
- Текущий счётчик, статус, последнее обслуживание.
- Выручка за 30 дней, средний ROI, тренд.
- Количество обслуживаний, суммарные дни простоя.

Нижняя зона (история обслуживаний):
- Таблица: дата, время, техник, счётчик игр, новых игр, тест игр, выручка, себестоимость, ROI, статус, комментарий, фото (3 иконки).
- Фильтры: диапазон дат.
- Клик на строку → детальная карточка обслуживания с тремя фото (просмотр + скачивание).

### 8.3. Журнал обслуживаний

Сводная таблица всех обслуживаний.

Фильтры: аппарат, диапазон дат, техник, город, маршрут, район, статус аппарата.

Колонки: дата, время, № аппарата, адрес, техник, новых игр, выручка, себестоимость, ROI, статус, комментарий.

Действия: удалить обслуживание (с подтверждением, автоматический пересчёт цепочки).

### 8.4. Графики

- Линейный график: динамика новых игр, ROI по датам.
- Выбор: один аппарат или все.
- Ось X — даты, ось Y — значения.

### 8.5. Аналитика по технику

Для каждого техника:
- Количество обслуживаний за период.
- Среднее количество аппаратов в день.
- Среднее количество игрушек на обслуживание vs другие техники.
- Динамика выручки по его аппаратам.
- Корреляция расхода игрушек и ROI.
- Количество аппаратов в простое.
- Количество забытых аппаратов.

### 8.6. Сводные отчёты (вкладки)

**Отчёт по игрушкам:** общий расход (наименование, количество, себестоимость, доля) + по аппаратам.

**Локации и маршруты:** дерево с показателями (количество аппаратов, обслуживаний, выручка, средний ROI) + таблица по маршрутам.

**Финансовый отчёт:** группировка по месяцам — период, количество обслуживаний, выручка, себестоимость, прибыль, ROI, новых игр. Итоговая строка. График выручки и себестоимости.

### 8.7. Экспорт в Excel (финансовый отчёт)

Месячный отчёт по кнопке «Экспорт Excel». Фильтры: город, маршрут, район.

Структура файла:
- Одна строка = один аппарат (только те, у которых было хотя бы одно обслуживание за месяц).
- Колонки: № аппарата, адрес, счётчик игр (последний), новых игр за период, тест игр за период, выручка, расход игрушек по видам (штуки), расход игрушек по видам (рубли).
- Итоговый блок внизу (суммы по всем колонкам).

### 8.8. Управление справочниками

Стандартные таблицы с CRUD (модальные окна для редактирования).

### 8.9. Аудит

Таблица `audit_log` с фильтрацией по: таблице, записи, пользователю, дате.

Хранение: 12 месяцев. Более старые — в архивную таблицу или удаление (node-cron, раз в месяц).

## 9. Безопасность

- Только HTTPS.
- JWT access token (1 час) + refresh token (7 дней, httpOnly cookie, rotation при использовании).
- bcrypt для паролей (salt 10).
- Валидация всех входных данных на сервере (Zod схемы из `packages/shared`).
- CSP заголовки, CSRF-защита (токен в заголовках).
- Presigned URLs для доступа к фото (TTL 1 час).

```typescript
// packages/client/src/store/db.ts

import Dexie, { Table } from 'dexie';

// Черновик обслуживания
interface DraftService {
  id?: number;                    // autoincrement
  localId: string;                // uuid, для идентификации
  machineNumber: number;
  machineAddress: string;
  serviceDate: string;            // ISO date
  serviceTime: string;            // HH:MM
  gameCounter: number;
  prizeCounter?: number;
  testGames: number;
  isOperational: boolean;
  comment?: string;
  toys: Array<{
    toyId: number;
    toyName: string;
    quantity: number;
    priceSnapshot: number;
  }>;
  photoCounter?: Blob;
  photoBefore?: Blob;
  photoAfter?: Blob;
  status: 'pending' | 'error';
  errors?: string[];
  createdAt: string;
}

// Кэш аппарата
interface CachedMachine {
  number: number;                 // primary key
  address: string;
  locationId: number;
  typeId: number;
  typeName: string;
  status: string;
  pricePerGame: number;
  hasPrizeCounter: boolean;
  minServiceDays?: number;
  maxServiceDays?: number;
  toys: Array<{
    toyId: number;
    name: string;
    price: number;
  }>;
  technicianIds: number[];
  updatedAt: string;
}

// Кэш последних обслуживаний (для тренда ROI и прошлых значений)
interface CachedServiceHistory {
  id?: number;
  machineNumber: number;          // index
  serviceDate: string;
  serviceTime: string;
  gameCounter: number;
  prizeCounter?: number;
  testGames: number;
  newGames?: number;
  revenue?: number;
  costOfToys?: number;
  roi?: number;
  isOperational: boolean;
}

// Кэш локаций
interface CachedLocation {
  id: number;
  name: string;
  parentId?: number;
  nodeType?: string;
  minServiceDays?: number;
  maxServiceDays?: number;
  updatedAt: string;
}

// Кэш игрушек
interface CachedToy {
  id: number;
  name: string;
  price: number;
  updatedAt: string;
}

// Метаданные синхронизации
interface SyncMeta {
  key: string;                    // primary key
  lastSyncAt: string;             // ISO datetime
}

class SlotManagerDB extends Dexie {
  drafts!: Table<DraftService>;
  machines!: Table<CachedMachine>;
  serviceHistory!: Table<CachedServiceHistory>;
  locations!: Table<CachedLocation>;
  toys!: Table<CachedToy>;
  syncMeta!: Table<SyncMeta>;

  constructor() {
    super('SlotManagerDB');
    this.version(1).stores({
      drafts:         '++id, localId, machineNumber, status, createdAt',
      machines:       'number, locationId, typeId, status',
      serviceHistory: '++id, machineNumber, serviceDate',
      locations:      'id, parentId',
      toys:           'id',
      syncMeta:       'key',
    });
  }
}

export const db = new SlotManagerDB();
```

## 14. Алгоритм пересчёта цепочки

```typescript
// apps/server/src/services/chainRecalc.ts

async function recalcChain(
  placementId: number,
  startDate: Date,
  tx: Transaction
): Promise<void> {

  // 1. Получить все обслуживания placement начиная с startDate
  const services = await tx
    .select()
    .from(serviceTable)
    .where(
      and(
        eq(serviceTable.placementId, placementId),
        gte(serviceTable.serviceDate, startDate)
      )
    )
    .orderBy(asc(serviceTable.serviceDate), asc(serviceTable.serviceTime))
    .for('update');  // SELECT FOR UPDATE

  // 2. Получить предыдущее обслуживание
  const prevService = await tx
    .select()
    .from(serviceTable)
    .where(
      and(
        eq(serviceTable.placementId, placementId),
        lt(serviceTable.serviceDate, startDate)
      )
    )
    .orderBy(desc(serviceTable.serviceDate), desc(serviceTable.serviceTime))
    .limit(1);

  let prev = prevService[0] ?? null;

  // 3. Пересчёт для каждого обслуживания
  for (const current of services) {
    const updates: Partial<Service> = {};

    updates.beforeServiceId = prev?.id ?? null;

    if (prev) {
      // Период в днях
      const daysDiff = differenceInDays(current.serviceDate, prev.serviceDate);
      updates.periodDays = Math.max(1, daysDiff);

      // Новые игры
      updates.newGames = current.gameCounter - prev.gameCounter - current.testGames;

      // Новые призы
      if (current.prizeCounter != null && prev.prizeCounter != null) {
        updates.newPrizes = current.prizeCounter - prev.prizeCounter;
      }

      // Выручка (только если аппарат работал в предыдущем периоде)
      if (prev.isOperational) {
        updates.revenue = (updates.newGames ?? 0) * current.pricePerGame;
      } else {
        updates.revenue = null; // период простоя
      }

      // ROI = выручка текущего / себестоимость предыдущего
      if (updates.revenue != null && prev.costOfToys && prev.costOfToys > 0) {
        updates.roi = updates.revenue / Number(prev.costOfToys);
      } else {
        updates.roi = null;
      }
    } else {
      // Первое обслуживание на placement
      updates.periodDays = null;
      updates.newGames = null;
      updates.newPrizes = null;
      updates.revenue = null;
      updates.roi = null;
    }

    await tx
      .update(serviceTable)
      .set(updates)
      .where(eq(serviceTable.id, current.id));

    prev = { ...current, ...updates };
  }
}
```

## 15. Стратегии Service Worker (Workbox)

| Маршрут | Стратегия | Описание |
|---|---|---|
| `/` и статические ассеты | CacheFirst | Оболочка PWA всегда из кэша |
| `/api/cache/*` | NetworkFirst | Справочники — сначала сеть, при ошибке кэш |
| `/api/sync` | NetworkOnly | Только сеть, оффлайн — в очередь |
| `/api/reports/*` | NetworkOnly | Отчёты только онлайн |
| Фото из MinIO | CacheFirst (TTL 24ч) | Фото кэшируются на 24 часа |

Background Sync очередь: `sync-queue` — черновики, которые не удалось отправить при первой попытке.

## 20. Критерии приёмки

1. PWA устанавливается на Android и iOS, запускается без адресной строки.
2. Оффлайн: техник выбирает аппарат из кэша, вводит данные, делает 3 фото, сохраняет черновик. Видит тренд ROI из кэша.
3. После синхронизации: данные в БД, фото в MinIO, пересчёт цепочки выполнен.
4. Некорректный счётчик → сервер возвращает ошибку → черновик остаётся с пометкой «ошибка».
5. Замена аппарата через админку → старый placement закрыт, новый открыт, история сохранена.
6. Забытые аппараты → баннер при открытии приложения.
7. Администратор: журнал обслуживаний, удаление записи с пересчётом, финансовый отчёт за месяц, экспорт Excel.
8. QR-код аппарата — генерируется PDF, сканируется камерой в форме обслуживания.
9. Аудит фиксирует все операции.

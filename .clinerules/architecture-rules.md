# Жёсткие правила архитектуры проекта Apix (Slot Manager)

Эти правила нельзя нарушать ни при каких изменениях:

1. Пересчёт цепочки обслуживаний (revenue, ROI, period_days) выполняется ТОЛЬКО на сервере. Клиент никогда не считает финансовые показатели сам, только показывает то, что прислал сервер (или предварительный расчёт через эндпоинт /api/preview).

2. Пересчёт цепочки никогда не пересекает границу одного placement (см. таблицу machine_placements). Замена физического аппарата на точке закрывает старый placement и создаёт новый — это нормальная операция, не баг.

3. Цена игры и цены игрушек на момент обслуживания — это "снапшот" (зафиксированное значение), записанный в таблицу service и toy_distribution. Изменение текущей цены в справочнике НЕ должно пересчитывать уже сохранённые старые обслуживания.

4. Все типы и схемы валидации должны браться из packages/shared, а не дублироваться отдельно в client и server.

5. Все Zod-схемы — единственный источник правды о форме данных. TypeScript-типы выводятся из них, а не пишутся отдельно вручную.

6. Любое изменение схемы базы данных — через миграции Drizzle, никогда вручную через прямые SQL-команды на работающей базе.

Если задача требует нарушить одно из этих правил — остановись и спроси у пользователя, прежде чем продолжать.

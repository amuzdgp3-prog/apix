# Правила управления контекстом

При поиске и чтении файлов в проекте ОБЯЗАТЕЛЬНО игнорировать:
- node_modules/
- dist/
- build/
- .pnpm-store/
- *.log
- *.lock
- *.min.js
- *.map
- Все файлы из .gitignore, если они попали в рабочую область

Никогда не читать целиком большие автосгенерированные файлы (например,
drizzle миграции) — только конкретные строки, если это необходимо.

Перед чтением файла, который может быть большим (>500 строк), сначала
оценить его размер и спросить пользователя.
## Правила для терминала (Windows PowerShell)

Я всегда запускаю команды в PowerShell, а не в bash. Поэтому:
- Для перехода в папку И выполнения команды использую ТОЛЬКО `;`, а не `&&`.
  ❌ `cd apps/server && npx tsc --noEmit`
  ✅ `cd apps/server; npx tsc --noEmit`
- Если нужно выполнить несколько команд последовательно — разделяю их через `;`.
- Длинные команды разбиваю на несколько вызовов `cd ...; ...` или использую полные пути.
- Не использую `2>&1` для перенаправления ошибок — PowerShell использует `*>&1`.
## Ограничения при записи файлов

Если файл превышает ~2500 токенов, не пытайся записать его одним вызовом write_to_file.
Вместо этого:
- Создай файл с базовой структурой через write_to_file.
- Добавляй остальные блоки через несколько вызовов replace_in_file.
- Всегда проверяй, не обрезался ли файл, и дописывай остаток.
## Правила для терминала (Windows PowerShell) — ВЫПОЛНЯТЬ НЕУКОСНИТЕЛЬНО

Я работаю исключительно в среде PowerShell на Windows. Bash-синтаксис (`&&`, `||`, `2>&1`) здесь недопустим и приведёт к поломке команд.

1.  **Разделитель команд:** всегда использовать `;` для последовательного выполнения.
    ❌ `cd server && npx tsc`
    ✅ `cd server; npx tsc`

2.  **Перенаправление ошибок:** всегда использовать `*>&1`.
    ❌ `npx tsc 2>&1`
    ✅ `npx tsc *>&1`

3.  **Работа с переменными:** для установки переменной окружения использовать `$env:VAR = "value"`.
    ❌ `EXPORT VAR=value`
    ✅ `$env:VAR = "value"`

4.  **Длинные команды:** разбивать на несколько логических блоков, каждый на новой строке или с чётким разделением `;`. Избегать однострочников длиннее 80 символов.

При любой проверке компиляции TypeScript (tsc) я НИКОГДА не полагаюсь на захват вывода из терминала.
Вместо этого я ОБЯЗАН:

1. Запустить компиляцию с перенаправлением ВСЕХ потоков в файл:
   `cd c:\Users\Evgeny\Projects\Apix\apps\server; npx tsc --noEmit *>&1 | Out-File -FilePath tsc_output.txt -Encoding UTF8`
2. Прочитать файл `tsc_output.txt` инструментом read_file.
3. Если файл пустой — ошибок нет. Если содержит текст — проанализировать ошибки.
4. После анализа удалить файл: `Remove-Item tsc_output.txt`.
## ЗАПРЕТ на временные скрипты и ручную работу с файлами через Node.js

- НИКОГДА не создавай временные JS-скрипты (например, `_tmp_write.js`, `fix_file.js`) для работы с файлами.
- НИКОГДА не используй `fs.writeFileSync` или `require("fs")` в таких скриптах.
- Для создания/редактирования файлов используй ТОЛЬКО предоставленные инструменты:
  - `write_to_file` — для новых или полной перезаписи файлов.
  - `replace_in_file` — для частичного редактирования существующих файлов.
- Если файл слишком большой для одной операции — разбей его на несколько вызовов `replace_in_file`.
# Правила управления контекстом и командами

## Правила для терминала (Windows PowerShell)
- Разделитель: ТОЛЬКО `;`. Запрещены `&&`, `||`.
  ❌ `cd server && npx tsc`
  ✅ `cd server; npx tsc`
- Перенаправление: ТОЛЬКО `*>&1`.
  ❌ `npx tsc 2>&1`
  ✅ `npx tsc *>&1`
- Переменные: `$env:PGPASSWORD = "secret"`

## Правило проверки TypeScript
- ВСЕГДА писать вывод в файл и читать его:
  `cd apps/server; npx tsc --noEmit *>&1 | Out-File -FilePath tsc_output.txt -Encoding UTF8`
- После проверки удалить файл: `Remove-Item tsc_output.txt`

## Запрет на временные скрипты
- НИКОГДА не создавать `_tmp_write.js`, `fix_file.js` и подобные.
- Использовать ТОЛЬКО `write_to_file` и `replace_in_file`.
- Если файл слишком большой — разбить на 2-3 вызова.

## Ограничения при записи файлов
- Если `write_to_file` обрезал файл — дополнить через `replace_in_file`.
- При 3 неудачных попытках — остановиться и сказать "Нужна ручная вставка".

## Игнорирование файлов и папок
- Игнорировать: `node_modules/`, `dist/`, `build/`, `.pnpm-store/`, `*.log`, `*.lock`.
- Перед чтением файла >500 строк оценить размер и спросить пользователя.
